# 一、初始阶段：HTTPS 协议推送 各类报错（前期问题汇总）

## 1.1 首次推送：大文件 + 网络中断报错

### 执行命令

```bash
git push -u origin main --force
```

### 报错信息

```
Enumerating objects: 679, done.
Counting objects: 100% (679/679), done.
Delta compression using up to 12 threads
Compressing objects: 100% (256/256), done.
error: RPC failed; curl 55 Recv failure: Connection was reset
send-pack: unexpected disconnect while reading sideband packet
Writing objects: 100% (677/677), 1.73 GiB | 9.66 MiB/s, done.
Total 677 (delta 399), reused 677 (delta 399), pack-reused 0 (from 0)
fatal: the remote end hung up unexpectedly
Everything up-to-date
```

### 问题原因

1. 项目体积过大（1.73GB），超出 Git 默认传输缓存上限；
2. 访问 GitHub 网络不稳定，远程主动断开连接；
3. `Everything up-to-date` 为**伪成功**，代码并未完整上传；
4. `--force` 为强制推送参数，新手慎用。

### 临时优化方案（大文件传输）

增大 Git 传输缓存，缓解大文件传输中断问题：

```bash
git config http.postBuffer 524288000
git config https.postBuffer 524288000
git config --global http.lowSpeedLimit 0
git config --global http.lowSpeedTime 999999
```

> 补充建议：超大工程建议**分批推送**，剔除视频、压缩包、大型资源文件。

## 1.2 彻底重置：删除 `.git` 目录，从零重建仓库

旧 Git 记录存在异常缓存，直接删除项目内隐藏文件夹 `.git`，清空所有历史记录，重新初始化仓库。

### 全套初始化命令（逐条执行）

```bash
# 1. 初始化全新本地 Git 仓库（仅首次执行）
git init

# 2. 配置全局提交身份（全电脑生效）
git config --global user.name "xx"
git config --global user.email "xx@qq.com"

# 3. 所有文件加入暂存区（. 代表当前目录全部文件）
git add .
```

### 执行 `git add .` 出现换行符警告

#### 警告内容

```
warning: in the working copy of 'xxx.uvprojx', LF will be replaced by CRLF the next time Git touches it
```

#### 说明

`LF` 为类 Unix 换行符，`CRLF` 为 Windows 换行符；**纯警告，非错误**，Git 自动统一格式，可直接忽略。

### 继续完成本地提交 & 关联远程仓库

```bash
# 4. 暂存区文件提交到本地仓库，-m 后跟提交备注（必填）
git commit -m "initial commit"

# 5. 关联 GitHub 远程仓库（HTTPS 地址）
git remote add origin https://github.com/xx/SimWheel.git

# 6. 再次配置大文件传输缓存
git config http.postBuffer 524288000
```

## 1.3 分支名不匹配，推送失败

### 原因

本地新建 Git 仓库默认分支为 `master`，命令指定推送到远程 `main` 分支，**分支名称不匹配**。

```bash

```

```
Counting objects: 100% (670/670), done.

Compressing objects: 100% (651/651), done.

Total 670 (delta 353), reused 0 (delta 0), pack-reused 0 (from 0)
remote: Resolving deltas: 100% (353/353), done.
 * [new branch]      master -> master
branch 'master' set up to track 'origin/master'.
```

✅ 代码成功推送到远程 `master` 分支。

### 衍生问题

GitHub 网页默认展示 `main` 分支，因此主页看不到新代码；

临时查看方式：访问 `https://github.com/xx/SimWheel/tree/master`；

优化目标：将本地、远程分支统一为 GitHub 默认分支 `main`。

## 1.4 分支重命名引发远程冲突

### 操作：本地分支重命名 master → main

```bash
git branch -m master main
```

终端标识变为 `(main)`，重命名成功。

### 推送新分支，首次冲突报错

#### 执行命令

```bash
git push -u origin main
```

#### 报错信息

```
! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'https://github.com/xx/SimWheel.git'
hint: Updates were rejected because the remote contains work that you do not have locally.
```

#### 原因

远程仓库原本存在 `main` 分支（旧代码），与本地新分支内容冲突，Git 禁止直接覆盖。

### 尝试安全强制推送，再次报错

```bash
git push origin main --force-with-lease
```



```
! [rejected]        main -> main (stale info)
error: failed to push some refs to 'https://github.com/xx/SimWheel.git'
```

原因：`--force-with-lease` 带安全校验，检测远程分支状态异常，拦截覆盖操作。

## 1.5 高危操作：误执行 `reset --hard`，本地文件错乱（重大踩坑）

### 执行命令（错误操作）

```bash
# 同步远程分支状态
git fetch origin --prune

# 高危命令：强制用远程旧代码覆盖本地所有文件
git reset --hard origin/main
```

### 后果

本地原始工程目录、文件结构全部被远程旧代码替换，文件错乱，看似 “文件丢失”。

## 1.6 HTTPS 协议 443 端口 & SSL 证书报错（核心网络问题）

### 报错 1：SSL 证书校验失败

```
fatal: unable to access 'xxx': SSL certificate OpenSSL verify result: unable to get local issuer certificate (20)
```

### 尝试修复方案（未根治）

```bash
# 切换为 Windows 系统证书库
git config --global http.sslBackend schannel
# 临时关闭 SSL 校验（不安全，仅应急）
git config --global http.sslVerify false
```

### 报错 2：443 端口连接超时

```bash
fatal: unable to access 'https://github.com/xx/SimWheel.git/':
Failed to connect to github.com port 443
```

### 问题定位

1. HTTPS 协议访问 GitHub 网络不稳定、存在丢包；
2. 本地网络拦截 443 端口，HTTPS 链路持续抽风；
3. `ping github.com` 可连通，但丢包严重，网络环境不适配 HTTPS。

### 最终决策

**彻底放弃 HTTPS 协议，改用 SSH 协议**

优势：连接更稳定、极少出现 443 端口问题、适配大工程长期开发。

# 二、进阶阶段：SSH 协议完整配置（根治网络问题）

## 2.1 生成 SSH 密钥对

### 执行命令

```bash
ssh-keygen -t ed25519 -C "xx@qq.com"
```

### 操作说明

执行后**一路回车**，无需设置密钥密码；

生成文件路径（Windows Git 默认路径）：

- 私钥：`~/.ssh/id_ed25519`（保密，不可泄露）
- 公钥：`~/.ssh/id_ed25519.pub`（用于绑定 GitHub）

## 2.2 启动 SSH 代理并加载密钥

```bash
# 启动 ssh-agent 代理
eval "$(ssh-agent -s)"

# 将私钥添加到代理
ssh-add ~/.ssh/id_ed25519
```

成功提示：`Identity added`

## 2.3 查看并复制公钥

```bash
cat ~/.ssh/id_ed25519.pub
```

### 关键注意事项

输出内容格式：`ssh-ed25519 AAAA......邮箱`

**整行全部复制**，不要只截取中间部分。

## 2.4 公钥绑定到 GitHub

1. 打开 GitHub 网页 → 右上角头像 → `Settings` → `SSH and GPG Keys`
2. 点击 `New SSH key` 新建密钥
3. 填写配置：
     - Title：自定义名称（例：MyPC）
     - Key：粘贴上一步复制的**完整公钥内容**
4. 保存完成绑定。

## 2.5 修改 Git 远程仓库地址（HTTPS → SSH）

### 执行命令

```bash
git remote set-url origin git@github.com:xx/SimWheel.git
```

### 地址替换规则

- 原 HTTPS 地址：`https://github.com/xx/SimWheel.git`
- 新 SSH 地址：`git@github.com:xx/SimWheel.git`

## 2.6 测试 SSH 连通性

```bash
ssh -T git@github.com
```

1. 首次连接提示 `Are you sure you want to continue connecting`，输入 `yes` 回车；

2. 成功提示：

     ```
     Hi xx! You've successfully authenticated
     ```

     

     ✅ SSH 协议配置完成，链路正常。

## 2.7 SSH 模式下推送代码（最终成功）

```bash
git push --force origin main
```

成功输出：`main -> main`

✅ 代码正常上传至 GitHub，网络问题彻底解决。

# 三、衍生问题排障 & 数据恢复（核心救命操作）

## 3.1 问题：`Everything up-to-date` 文件修改后无更新

### 现象

本地修改文件后执行推送，提示 `Everything up-to-date`，远程无变化。

### 原因

仅执行 `git add .` 把文件加入暂存区，**未执行 `git commit` 生成本地提交**，Git 识别不到文件变更。

### 原理补充

Git **不是网盘**，不会自动同步文件，必须走完完整三步流程。

## 3.2 误区：误判文件丢失（`tree` 命令 + reset 快照问题）

### 现象

执行 `tree` 查看目录，发现 `Docs`、`CMSIS` 等目录消失，误以为工程文件损坏、数据丢失。

### 真实原因

1. `tree` 命令默认展示规则问题，并非文件物理删除；
2. 此前执行 `git reset --hard`，工作区被切换到**旧版本提交快照**，只是切换了工程历史状态。

## 3.3 核心救命：`git reflog` 恢复被覆盖的本地文件

### 功能说明

`git reflog` 是 Git 操作日志（时光机），记录所有分支切换、reset、提交等历史动作，可找回丢失的版本 / 文件。

### 步骤 1：查看历史操作日志

```bash
git reflog
```

日志输出参考：

```
fc42484 (HEAD -> main, origin/main, origin/HEAD) HEAD@{0}: commit: 更新STM32工程文件，更新文档，删除冗余资源
3c1279a HEAD@{1}: reset: moving to origin/main
0b1d4a6 (origin/master) HEAD@{2}: Branch: renamed refs/heads/master to refs/heads/main
0b1d4a6 (origin/master) HEAD@{4}: commit (initial): 重新配置文件大小，更新文档，更新工程文件，删除冗余资源
```

- `0b1d4a6`：原始完整工程的提交哈希值（目标恢复节点）

### 步骤 2：错误写法（踩坑记录）

```bash
git reset --hard 【0b1d4a6】  # 错误：多余中英文括号，命令识别失败
```

### 步骤 3：正确恢复命令（无多余符号，直接使用哈希值）

```bash
git reset --hard 0b1d4a6
```

✅ 执行后，本地目录、文件结构**完全恢复为原始状态**。

## 3.4 Git 核心本质理解

Git 是**时间快照系统**：

- 每一次 `commit` 提交，都会保存**整个工程的完整状态快照**；
- 并非只保存文件修改部分，因此可以任意回滚到任意历史快照。

# 四、工程规范配置（.gitignore、提交规范、版本管理）

## 4.1 `.gitignore` 忽略文件配置

### 问题背景

前期误将编译产物、本地缓存文件上传至仓库：`hex`、`Output`、`Objects`、`uvoptx`、`uvguix` 等。

### 文件分类判定

1. **必须忽略（禁止上传）**：编译输出文件、Keil 本地配置、临时缓存文件；
2. **保留上传（项目资产）**：`Docs`、`Hardware`、Images、Logs、源码等核心工程文件。

### 重要知识点

`.gitignore` **仅对未跟踪的新文件生效**，已经 `commit` 提交到仓库的文件，无法通过该配置自动删除。

## 4.2 `git push --force` 强制推送解读

- 作用：使用**本地仓库历史快照，强制覆盖远程仓库**；
- 适用场景：版本回滚（reset）、仓库历史重构、仓库整理；
- 风险提醒：多人协作项目严禁使用，会覆盖他人代码；单人私有仓库可按需使用。

## 4.3 标准 Git 开发工作流（日常更新固定流程）

每次修改代码后，统一执行以下三条命令：

```bash
# 1. 变更文件加入暂存区
git add .
# 2. 生成本地提交，备注修改内容
git commit -m "修改说明"
# 3. 推送到远程仓库
git push
```

## 4.4 Commit 提交信息规范

### 不推荐（模糊、无意义）

```
fix
test
更新
111
```

### 推荐（语义清晰，便于版本追溯）

```
添加USB HID驱动
修复编码器抖动问题
重构STM32目录结构
更新README文档
```

## 4.5 Tag / Release 版本管理概念

1. `commit`：开发过程中的普通存档点，记录每一次代码变更；
2. `Tag/Release`：正式版本标记，用于对外发布，命名示例：`v0.1`、`v0.5`、`v1.0`。

## 6.2 高危命令避坑清单（重点牢记）

1. `git reset --hard 哈希/分支`：强制覆盖本地文件，无备份极易丢失数据，非必要不使用；
2. `git push --force`：强制覆盖远程代码，多人协作绝对禁止；
3. 删除 `.git` 目录：清空所有 Git 历史记录，会丢失版本追溯能力。

```bash
# 3. 回滚到指定历史版本
git reset --hard 哈希值

# 4. 查看远程仓库地址
```
